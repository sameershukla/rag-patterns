# RAG Patterns — Real-World Examples

A collection of **Retrieval Augmented Generation (RAG)** implementations built
from scratch in pure Python. No LangChain. No abstractions. Each example solves
a real business problem and is written to be read and understood line by line.

Built as part of the *"From Tokens to Agents"* AI learning series.
Series 2 — RAG & Semantic Search.

## What is RAG?

RAG solves four fundamental limitations of every LLM:

| Problem | What it means |
|---|---|
| Knowledge cutoff | LLM knows nothing after its training date |
| No private data | Your internal docs were never in its training |
| Hallucination | LLM confidently makes up answers when it doesn't know |
| Context window | You can't dump your entire knowledge base into every prompt |

**RAG solves all four** by retrieving only the most relevant documents at query
time and injecting them into the prompt — so the LLM answers from your real,
current, private knowledge instead of guessing.

## The pattern every example follows

All examples are split into three files using the same two-phase RAG architecture:

```
PHASE A — INDEXING (offline, run once)
─────────────────────────────────────────────────
indexer.py  →  Load docs → Embed → Store in FAISS → Save to disk

PHASE B — QUERYING (online, runs per request)
─────────────────────────────────────────────────
retriever.py  →  Load index → Embed query → Search → Return top-K
app.py        →  Retrieve → Augment prompt → Call LLM → Return answer
```

Once you understand this pattern in one example, you understand it in all of them.

## Examples

### 1. `rag-customer-support` — E-commerce Support Bot

**Business problem:** An e-commerce store's LLM-powered support bot needs to
answer customer questions accurately based on the store's actual policies —
not from general training knowledge that may be outdated or wrong.

**What it does:** Indexes store policy documents (returns, shipping, damaged
items, cancellations, gift cards) into a FAISS vector index. When a customer
asks a question, retrieves the most relevant policy and passes it to Claude
to generate a grounded, accurate answer.

**Key concepts demonstrated:**
- Phase A vs Phase B pipeline split
- FAISS `IndexFlatIP` with normalised vectors
- Score threshold guard to prevent answering irrelevant questions
- Prompt augmentation with retrieved context
- Grounding the LLM with "answer ONLY from the context below"

See [`rag-customer-support/README.md`](./rag-customer-support/README.md) for
full setup instructions and a line-by-line code explanation.

## Getting started

### Prerequisites

- Python 3.10 or higher
- An Anthropic API key — get one at https://console.anthropic.com
- ~$5 in API credits (each test run costs fractions of a cent)

### Running any example

Every example follows the same three steps:

```bash
# 1. Go into the example folder
cd rag-customer-support

# 2. Install dependencies
pip install -r requirements.txt

# 3. Set your API key
export ANTHROPIC_API_KEY=sk-ant-your-key-here

# 4. Build the index (run once)
python indexer.py

# 5. Run the app
python app.py
```

Full setup details are in each example's own `README.md`.

## Is it safe to push this repo publicly?

Yes. Your API key is never written in any file — it is set as a terminal
environment variable and read automatically by the Anthropic SDK. However,
each example generates local files that should not be committed:

Each subfolder's `.gitignore` excludes:
```
index.faiss       # generated by indexer.py
documents.json    # generated by indexer.py
.env              # if you use a .env file for your key
__pycache__/
venv/
```

Never store your API key inside any `.py` file or commit a `.env` file to
a public repository.
